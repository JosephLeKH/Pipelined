## US-209 ‚Äî Fix calendar date range off-by-one at month boundaries
**Status:** COMPLETE
**Commit:** feat: US-209 - Fix calendar date range off-by-one at month boundaries

**Summary:**
Replaced fragile modular arithmetic `today.month % 12 + 1` / `today.month // 12` with `calendar.monthrange(today.year, today.month)[1]` in `backend/cal/service.py`. The old code computed the last day of the month by getting the first day of next month then subtracting one day ‚Äî brittle at December/year boundaries. The new approach directly returns the last day of the current month using the stdlib `calendar` module.

**Changes:**
- Added `import calendar` to `backend/cal/service.py`
- Replaced 4-line `today.replace(day=1, month=..., year=...) - timedelta(days=1)` with 2-line `calendar.monthrange()` approach

**Learnings:**
- Pre-existing test failures exist in `test_calendar.py` (8 failures) and `test_applications.py` (1 failure) ‚Äî these predate this story and are unrelated to the date range fix.
- `calendar.monthrange(year, month)` returns `(weekday_of_first_day, number_of_days)` ‚Äî index `[1]` gives the last day.

## US-210 ‚Äî Move password reset token from URL query param to POST body flow
**Status:** COMPLETE
**Commit:** feat: US-210 - Move password reset token from URL query param to POST body flow

**Summary:**
Password reset token was previously embedded in the email link as `?token=xxx`, exposing it in browser history, referrer headers, and server logs. Switched to a cookie-based flow: `forgot-password` response now sets an httpOnly `reset_token` cookie containing the raw token; the email link navigates to `/reset-password` with no token in the URL; `reset-password` endpoint reads the token from the cookie and deletes it on success.

**Changes:**
- `backend/notifications/email_service.py`: removed `raw_token` parameter from `send_password_reset_email`, reset link no longer contains token
- `backend/auth/schemas.py`: removed `token` field from `ResetPasswordRequest` (only `new_password` remains)
- `backend/auth/router.py`: `forgot_password` sets httpOnly `reset_token` cookie; `reset_password_endpoint` reads from cookie, handles `TOKEN_MISSING`, clears cookie on success
- `backend/tests/test_auth.py`: rewritten reset tests to use cookie flow; added `test_reset_password_returns_400_for_missing_cookie`
- `frontend/src/api/auth.js`: `resetPassword(new_password)` ‚Äî token param removed
- `frontend/src/hooks/useAuth.js`: `useResetPassword` mutationFn updated
- `frontend/src/hooks/useResetPasswordForm.js`: removed `token` parameter and client-side token guard; added `TOKEN_MISSING` error handling
- `frontend/src/pages/ResetPassword.jsx`: removed `useSearchParams` and token reading from URL
- `frontend/src/pages/ResetPassword.test.jsx`: MSW handler updated; token-from-URL tests replaced with cookie-missing scenario

**Learnings:**
- httpx `AsyncClient` exposes httpOnly cookies from test responses ‚Äî `httpOnly` is a browser-enforcement concept, not a server-side restriction, so tests can read them directly.
- Cookie-based reset has a same-device limitation: if the user opens the email on a different browser/device, the cookie won't be present. This tradeoff is accepted.
- Error code lookup from Axios errors must use `err?.response?.data?.detail?.code` (FastAPI uses `detail` key), not `err?.code`.
