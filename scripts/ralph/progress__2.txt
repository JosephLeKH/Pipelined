## US-041 — Fix extension save queue to survive individual save errors
**Status:** COMPLETE
**Commit:** feat: US-041 - Fix extension save queue to survive individual save errors

### Changes
- **`extension/background/background.js`**: Rewrote `handleSave` to attach `.catch()` directly on `executeSave(payload)` before assigning to `saveQueue`. This ensures `saveQueue` always resolves even when `executeSave` throws (network error, etc.), so subsequent saves are never silently dropped. The catch handler calls `console.error("Save failed:", err)` and returns `{ status: "error", message: "Save failed — try again" }`. Exported `handleSave` for testability.
- **`extension/content/content.js`**: Updated `showBannerError` banner text from "Save failed. Try again?" to "Save failed — try again" to match acceptance criteria.
- **`extension/tests/test_background.js`**: New test file with 4 tests covering: queue continues after throw, `console.error` is called, error status returned, and `chrome.storage.local.set` not called on failure.

### Learnings
- The subtle bug: `saveQueue = saveQueue.then(() => executeSave(payload))` — if `executeSave` throws, `saveQueue` becomes a rejected promise. Subsequent `.then()` calls on a rejected promise skip the callback, so all future saves silently drop. The fix: `.catch()` must be on the inner promise (`executeSave(payload).catch(...)`) so `saveQueue` itself always resolves.
- ES module import caching means the shared `saveQueue` state persists across all tests in the file — `beforeEach` must re-mock after `clearAllMocks` wipes implementations, but module-level state (the queue itself) remains valid since all prior tasks resolve.

### Bug Fixes (pre-existing issues surfaced by running full test suite)
- **`backend/cal/schemas.py`**: Removed `model_config = ConfigDict(strict=True)` from `EventCreate` and `EventUpdate` — Pydantic v2 strict mode rejects ISO date strings from JSON bodies, breaking all calendar creation endpoints.
- **`backend/cal/service.py`**: Converted `dt.date` → `dt.datetime` (UTC midnight) for BSON storage; stored `dt.time` as ISO string. Added `_date_to_datetime()` helper. Fixed date range filter to use `datetime` in MongoDB queries.
- **`backend/cal/schemas.py`** (`EventResponse.from_doc`): Converts stored `datetime` back to `date`, and stored time string back to `dt.time`.
- **`backend/tests/test_jobs.py`** (`_insert_listing`): Added `url_hash` computation to prevent `DuplicateKeyError` on the unique `url_dedup` index when inserting multiple listings with the same (absent) `url_hash`.
- **`backend/tests/test_calendar.py`** (`test_list_events_caps_at_500_documents`): Changed `base_date` from `dt.date` to `dt.datetime` for direct MongoDB inserts.

### Learnings
- **`ConfigDict(strict=True)` breaks JSON APIs**: Pydantic v2 strict mode doesn't coerce ISO date strings to `datetime.date` even in JSON parsing mode. FastAPI API schemas should not use strict mode unless they use custom JSON coercion.
- **BSON only encodes `datetime.datetime`, not `datetime.date`**: Always convert `dt.date` → `dt.datetime` before inserting into MongoDB; convert back in `from_doc()`.
- **pytest-asyncio `loop_scope` override**: Individual `@pytest.mark.asyncio` decorators override module-level `pytestmark`, dropping `loop_scope="session"`. Motor captures the event loop on first operation; mismatched loops cause "Future attached to a different loop" errors. Remove all bare `@pytest.mark.asyncio` decorators when using module-level `pytestmark`.
- **Rate limiter in tests**: `@limiter.limit()` route decorators enforce limits even without `SlowAPIMiddleware`. Set `limiter.enabled = False` in the `app` fixture to disable globally during tests.

**Learnings:**
- pydantic-settings v2 supports `@field_validator(mode="before")` on list fields to handle non-JSON env var formats (comma-separated strings). The validator receives the raw env value as a `str` and must return the parsed `list`.
- SlowAPI's `Limiter(key_func=...)` accepts any callable `(Request) -> str`. The single `key_func` is shared across all `@limiter.limit()` decorators — no per-decorator override needed.
- Tests that call `get_client_ip` directly (not through the HTTP stack) cleanly bypass the disabled limiter without needing HTTP client overhead.
