## US-041 — Fix extension save queue to survive individual save errors
**Status:** COMPLETE
**Commit:** feat: US-041 - Fix extension save queue to survive individual save errors

### Changes
- **`extension/background/background.js`**: Rewrote `handleSave` to attach `.catch()` directly on `executeSave(payload)` before assigning to `saveQueue`. This ensures `saveQueue` always resolves even when `executeSave` throws (network error, etc.), so subsequent saves are never silently dropped. The catch handler calls `console.error("Save failed:", err)` and returns `{ status: "error", message: "Save failed — try again" }`. Exported `handleSave` for testability.
- **`extension/content/content.js`**: Updated `showBannerError` banner text from "Save failed. Try again?" to "Save failed — try again" to match acceptance criteria.
- **`extension/tests/test_background.js`**: New test file with 4 tests covering: queue continues after throw, `console.error` is called, error status returned, and `chrome.storage.local.set` not called on failure.

### Learnings
- The subtle bug: `saveQueue = saveQueue.then(() => executeSave(payload))` — if `executeSave` throws, `saveQueue` becomes a rejected promise. Subsequent `.then()` calls on a rejected promise skip the callback, so all future saves silently drop. The fix: `.catch()` must be on the inner promise (`executeSave(payload).catch(...)`) so `saveQueue` itself always resolves.
- ES module import caching means the shared `saveQueue` state persists across all tests in the file — `beforeEach` must re-mock after `clearAllMocks` wipes implementations, but module-level state (the queue itself) remains valid since all prior tasks resolve.

## US-043 — Add result limit to calendar list_events and validate date range
**Status:** COMPLETE
**Commit:** feat: US-043 - Add result limit to calendar list_events and validate date range

### Changes
- **`backend/cal/service.py`**: Added `MAX_CALENDAR_EVENTS = 500` and `MAX_DATE_RANGE_DAYS = 366` module-level constants. Replaced `to_list(length=None)` with `to_list(length=MAX_CALENDAR_EVENTS)` in `list_events`.
- **`backend/cal/router.py`**: Imported `MAX_DATE_RANGE_DAYS` from service. Added `INVALID_DATE_RANGE_DETAIL` constant. Added guard in `list_events` handler: if `date_from` and `date_to` are both provided and `(date_to - date_from).days > MAX_DATE_RANGE_DAYS`, raises HTTP 400 with code `INVALID_DATE_RANGE`.
- **`backend/tests/test_calendar.py`**: Added 3 new tests: `test_list_events_caps_at_500_documents` (bulk-inserts 501 docs directly via `get_collection`, asserts response has 500), `test_list_events_returns_400_for_date_range_over_366_days` (367-day range), `test_list_events_accepts_date_range_of_exactly_366_days` (boundary, expects 200).

### Learnings
- Motor's `to_list(length=N)` silently caps the result — it doesn't raise if fewer docs exist. This makes it safe to always pass a cap.
- For the 500-doc cap test, using `get_collection("calendar_events").insert_many(docs)` in the test avoids 501 HTTP round-trips while still hitting a real MongoDB (no mocking). `get_collection()` raises if db is None, which satisfies type safety better than accessing `db[...]` directly.
- The AC said "mock returns 501 docs" but the project rule is "never mock the database". The real-DB bulk insert approach tests the same behaviour correctly.
- Pyright on this machine has pre-existing errors (unresolved imports, Python version for `|` union syntax) — no new errors introduced.
