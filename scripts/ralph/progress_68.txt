## US-678: Add aria-current='date' to today's calendar cell
**Status:** COMPLETE
**Commit:** 4608d8e

**Summary:** Added `aria-current={isToday ? "date" : undefined}` to the `role="button"` div in `DayCell` in `CalendarGrid.jsx`. When `isToday` is true the attribute is set to `"date"`; when false, passing `undefined` causes React to omit the attribute entirely from the DOM.

**Learnings:** React omits attributes with `undefined` values, so `aria-current={condition ? "date" : undefined}` is the idiomatic way to conditionally include an ARIA attribute without setting it to a falsy string like `"false"`.

## US-681: Fix Settings.jsx tab aria-controls WCAG violation
**Status:** COMPLETE
**Commit:** e1e2a49

**Summary:** Settings.jsx had `aria-controls={"panel-" + id}` on each of the 11 tab buttons, but only the active panel existed in the DOM (with `id={"panel-" + activeSection}`). Inactive tabs' `aria-controls` pointed to non-existent elements — a WCAG failure. Fixed by introducing a module-level constant `SETTINGS_PANEL_ID = "settings-panel"` and using it everywhere: all tab buttons now share `aria-controls={SETTINGS_PANEL_ID}`, and the `<main>` tabpanel always has `id={SETTINGS_PANEL_ID}`.

**Learnings:** For a single-panel tab UI where only one section is displayed at a time, using a single stable `aria-controls` target ID is the correct WCAG pattern. Each tab button should still have a unique `id` (for `aria-labelledby` on the panel), but since `renderSection` drives which content appears and the `<main>` element always exists, one stable panel ID cleanly satisfies the ARIA tab pattern without requiring all panels to be rendered and hidden.

## US-682: Add aria-hidden to EmptyState icon
**Status:** COMPLETE
**Commit:** 67ab492

**Summary:** `EmptyState.jsx` renders an icon via `<Icon className="..." />` on the ternary branch that handles the `icon` prop. Added `aria-hidden="true"` to this render so screen readers skip the decorative SVG and rely solely on the `title` and `description` text props for context.

**Learnings:** When an icon component is passed as a prop and used decoratively alongside descriptive text, `aria-hidden="true"` must be applied at the call site. The change is a single attribute addition on the JSX element — no structural refactoring needed.

## US-679: Wrap CalendarGrid weekday headers in abbr elements
**Status:** COMPLETE
**Commit:** 327d081

**Summary:** `CalendarGrid.jsx` was rendering short weekday abbreviations (Sun, Mon, etc.) as plain text inside `<div>` elements. Added `WEEK_DAYS_FULL` constant to `constants.js` with the 7 full weekday names (Sunday–Saturday). Updated the weekday header row to use `WEEK_DAYS.map((d, i) => ...)` and wrap each abbreviation in `<abbr title={WEEK_DAYS_FULL[i]}>{d}</abbr>`. Build passed successfully.

**Learnings:** Using `<abbr title="full name">abbr</abbr>` is the correct HTML pattern for exposing screen-reader-friendly full text for abbreviations. Pairing the short `WEEK_DAYS` array with a companion `WEEK_DAYS_FULL` array indexed by position keeps the mapping explicit, zero-magic, and consistent with the project's no-magic-values rule. The index approach avoids an object/Map lookup while staying readable.

## US-680: Fix EventChip BookOpen icon aria usage in CalendarGrid
**Status:** COMPLETE
**Commit:** 25e43c8

**Summary:** `CalendarGrid.jsx` `EventChip` had `<BookOpen aria-label="Has prep data" />` on the prep-data indicator icon. An `aria-label` on an SVG inside an already-labeled interactive button duplicates and confuses screen reader announcements. Changed `aria-label` to `aria-hidden="true"` and added `<span className="sr-only">Has prep notes</span>` as a sibling inside a React Fragment so screen readers still get the textual cue without reading the SVG directly.

**Learnings:** For decorative icons inside interactive elements that already carry an `aria-label`, the correct pattern is to hide the icon (`aria-hidden="true"`) and include any additional context as an `sr-only` text node. This avoids double-announcements while keeping the information available to assistive technology.

## US-683: Add screen-reader captions to AnalyticsCharts tables
**Status:** COMPLETE
**Commit:** 4e1b3ea

**Summary:** `AnalyticsCharts.jsx` had two data tables — `AnalyticsTagsTable` (tag offer rates) and `ConversionRatesTable` (conversion rates by stage) — with no `<caption>` elements. Screen readers could not identify each table's purpose. Added `<caption className="sr-only">Tag offer rates</caption>` as the first child of the tags table and `<caption className="sr-only">Conversion rates by stage</caption>` as the first child of the conversion rates table. Both tables already had `aria-label` attributes; the `<caption>` adds a visible-to-AT caption that satisfies WCAG SC 1.3.1.

**Learnings:** `<caption>` is the semantically correct way to label a `<table>` for screen readers; `aria-label` on the table element is a secondary technique. Adding both is valid and reinforces accessibility. Using `className="sr-only"` (Tailwind's visually-hidden utility) hides the caption from sighted users while keeping it in the accessibility tree. The caption must be the first child of `<table>`, before `<thead>`.
